# 通用增长指标接入与优化模板

适用对象：App、小游戏、SaaS、内容产品、电商、本地服务、工具型产品、ToB 线索业务。

目标：把业务从“看一堆数字”推进到“用指标定位增长断点，并知道应该改什么、怎么验证、何时继续或停止”。

## 1. 使用方式

复制本模板后，先填这 6 项：

```text
业务名称：
业务类型：
当前阶段：冷启动 / 新版本发布 / 已有稳定流量 / 投放学习 / 放量
核心增长问题：
当前主要渠道：
当前最重要的 1-3 个决策：
```

推荐一句话写清楚本轮增长命题：

```text
哪一种用户来源 / 承诺 / 入口 / 体验，能带来完成关键行为、重复使用，并产生业务价值的用户？
```

示例：

```text
哪一种广告素材和落地页承诺，能带来完成首次下单并在 7 天内复购的用户？
哪一种 onboarding 入口，能带来完成首次创建项目并在次周继续使用的团队？
哪一种内容 hook，能带来观看完成、点击注册并次日回访的用户？
```

## 2. 接入方式

接入顺序按“先能判断，再追求完整”推进。

### 2.1 最小接入

先保证能回答 4 个问题：

| 问题 | 必要数据 | 常见来源 |
| --- | --- | --- |
| 用户从哪里来 | source / medium / campaign / creative_id / referrer | 广告平台、UTM、应用商店、短视频平台、SEO |
| 用户是否进入核心体验 | activation event | App/Web 埋点、后端事件 |
| 用户是否完成关键行为 | conversion event | 后端订单、支付、表单、内容发布、游戏对局、预约 |
| 用户是否回来或复购 | retention / repeat event | 登录、打开、复购、再次创建、再次使用 |

最小事件建议：

| 事件 | 触发时机 | 必备参数 |
| --- | --- | --- |
| `session_start` 或 `app_open` | 用户打开产品 | `user_id`、`anonymous_id`、`source`、`campaign`、`creative_id` |
| `activation_start` | 用户开始核心流程 | `entry_source`、`variant_id`、`device`、`locale` |
| `activation_complete` | 用户完成首次关键体验 | `duration_sec`、`result`、`step_count`、`variant_id` |
| `conversion` | 用户完成业务关键行为 | `value`、`currency`、`plan`、`sku`、`order_id` |
| `repeat_action` | 用户第二次或多次使用 | `days_since_first_seen`、`repeat_index`、`source` |
| `retention_return` | 用户跨天回访 | `days_elapsed`、`previous_action_count`、`cohort_date` |

### 2.2 推荐接入架构

| 层 | 接入内容 | 用途 |
| --- | --- | --- |
| 客户端事件 | 页面浏览、按钮点击、关键流程开始/完成、错误、性能 | 判断体验断点和入口效果 |
| 服务端事件 | 注册、订单、支付、发货、审核、订阅、退款 | 作为业务结果的权威来源 |
| 渠道数据 | 广告花费、曝光、点击、安装、素材、关键词、国家 | 判断获客效率和素材质量 |
| 商店/平台数据 | store visitors、installers、retained installers、评分、评论 | 判断商店页和平台分发 |
| 用户维度 | 新老用户、国家、语言、设备、会员等级、行业、渠道 | 做 cohort 和分群诊断 |
| 数据仓库或 BI | 每日聚合、cohort、漏斗、留存、LTV | 稳定复盘和跨源对齐 |

### 2.3 数据字段规范

每条关键事件尽量带这些字段：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| `event_name` | 事件名 | `activation_complete` |
| `event_time` | 事件发生时间 | `2026-06-25T10:00:00+08:00` |
| `user_id` | 登录用户 ID，可能为空 | `u_123` |
| `anonymous_id` | 未登录用户 ID | `anon_abc` |
| `session_id` | 会话 ID | `s_456` |
| `source` | 来源 | `youtube`、`google`、`organic` |
| `medium` | 媒介 | `shorts`、`paid`、`search` |
| `campaign` | 活动 | `launch_test_20260625` |
| `creative_id` | 素材或页面版本 | `hook_a_matchup_question` |
| `variant_id` | 实验版本 | `landing_b` |
| `entry_source` | 产品内入口 | `home_cta`、`recommendation` |
| `country` / `locale` | 国家和语言 | `US`、`zh-CN` |
| `device` / `platform` | 设备和平台 | `android`、`ios`、`web` |
| `value` | 金额或业务价值 | `9.99` |
| `result` | 行为结果 | `success`、`failed`、`cancelled` |

## 3. 指标选择方法

不要从“能拿到什么数据”开始，而是从“这个数字会改变什么决策”开始。

### 3.1 指标分层

| 层级 | 作用 | 数量建议 | 示例 |
| --- | --- | ---: | --- |
| 北极星指标 | 表示业务真正增长 | 1 | 有价值活跃用户、完成订单用户、付费团队数 |
| 主要 KPI | 决定本阶段 go / hold / stop | 1-3 | 激活率、转化率、D1 留存、CAC |
| 驱动指标 | 解释 KPI 为什么变化 | 每个 KPI 1-2 | 首屏点击、关键步骤完成、素材点击率 |
| 诊断维度 | 定位问题在哪一类用户/来源/设备 | 按需 | source、creative_id、country、device |
| 护栏指标 | 防止为了提升 KPI 伤害质量 | 1-2 | 退款率、投诉率、崩溃率、退订率 |

### 3.2 选择标准

一个指标应该满足：

| 标准 | 判断问题 |
| --- | --- |
| 能代表目标 | 它是否真的说明业务变好了？ |
| 能指导动作 | 它变化后，我们知道要改渠道、产品、定价还是服务吗？ |
| 能被团队影响 | 本团队是否能通过 1-2 周动作影响它？ |
| 能稳定测量 | 是否每天或每周都能用同一口径拿到？ |
| 不容易被误刷 | 提升它是否可能掩盖质量、信任或留存问题？ |

### 3.3 常用指标模板

| 业务类型 | 北极星 | 激活指标 | 转化指标 | 留存指标 | 护栏 |
| --- | --- | --- | --- | --- | --- |
| App / 工具 | 每周完成核心任务用户 | 首次完成核心任务率 | 订阅 / 购买率 | D1 / D7 / W4 | 崩溃率、退款率 |
| SaaS | 活跃付费团队数 | 首次完成关键工作流 | 试用转付费 | 次周团队活跃 | churn、工单量 |
| 电商 | 有效订单用户数 | 商品详情到加购 | 支付成功率 | 7/30 天复购 | 退款率、履约失败 |
| 内容产品 | 有效观看 / 阅读用户 | 完播 / 阅读完成 | 注册 / 订阅 | 次日回访 | 低质量点击、取关 |
| 本地服务 | 有效预约 / 到店用户 | 表单完成率 | 预约成功率 | 二次预约 | 取消率、差评 |
| 游戏 / 互动产品 | 完成核心循环用户 | 首局完成率 | 付费 / 广告变现 | D1 / D7 | 卡顿、广告过载 |

## 4. 指标数据选择

### 4.1 数据优先级

当多个来源冲突时，按这个优先级判断：

| 数据问题 | 首选来源 | 次选来源 |
| --- | --- | --- |
| 订单、支付、退款 | 后端 / 支付系统 | 客户端埋点 |
| 曝光、点击、花费 | 广告平台 / 平台后台 | UTM 聚合 |
| 应用商店访问、安装、留存 | Play Console / App Store Connect | GA / MMP |
| 产品内行为 | 客户端 + 服务端事件 | 前端日志 |
| 留存 cohort | 数据仓库 / GA cohort / 平台留存 | 单日事件比值 |
| 用户反馈 | 客服 / 评论 / 调研 | 社媒截图 |

### 4.2 不能混用的口径

| 容易混淆 | 正确处理 |
| --- | --- |
| 点击率高 = 业务好 | 点击率只代表 hook，必须看激活、转化、留存 |
| `return_event / active_users` = D1 留存 | 这只是回访信号，D1 必须按 cohort 算 |
| 安装数 = 商店页转化好 | 还要看 visitors、source、country、retained installers |
| 客户端支付点击 = 真实收入 | 收入以服务端或支付系统为准 |
| 单日涨跌 = 趋势 | 小样本至少看 3-7 天或 cohort |

### 4.3 数据质量检查

每次复盘前先检查：

- [ ] 关键事件是否都有数据。
- [ ] 分母是否正确，例如新用户、活跃用户、访问用户还是安装用户。
- [ ] 时间窗口是否一致，例如自然日、UTC、业务时区。
- [ ] 是否区分新用户和老用户。
- [ ] 是否排除内部测试、机器人、员工账号。
- [ ] 渠道参数是否丢失或被覆盖。
- [ ] 事件是否重复上报。
- [ ] 跨端用户是否能合并。

## 5. 修改目标

修改目标不是“把指标变好”，而是明确要改哪一层。

### 5.1 指标到修改目标

| 指标异常 | 可能问题 | 修改目标 | 常见动作 |
| --- | --- | --- | --- |
| 曝光高，点击低 | 外部承诺不清楚 | 改素材 hook | 首图、标题、短视频前 3 秒、广告文案 |
| 点击高，安装/注册低 | 落地页或商店页不匹配 | 改落地页承诺 | 首屏、截图、价格、信任元素、加载速度 |
| 注册/打开高，激活低 | 用户不知道下一步 | 改 onboarding / 首屏 CTA | 默认推荐、引导、主按钮、减少配置 |
| 激活高，转化低 | 价值感或价格不成立 | 改 offer / pricing / proof | 免费试用、套餐、案例、优惠 |
| 首次完成高，二次使用低 | 没有回访理由 | 改循环和召回 | daily task、提醒、推荐、订阅内容 |
| 留存高，收入低 | 变现入口弱 | 改价值捕获 | 付费点、广告位置、升级提示 |
| 收入高，投诉/退款高 | 质量或承诺过度 | 改护栏 | 降低夸张承诺、补说明、修质量 |

### 5.2 目标写法

每个修改目标用这个格式：

```text
为了提升 [主指标]，我们要修改 [具体触点]，因为当前证据显示 [断点]。
成功标准是 [指标阈值]，护栏是 [不能恶化的指标]，观察窗口是 [时间]。
```

示例：

```text
为了提升 activation_complete / session_start，我们要修改首页主 CTA 和默认推荐，
因为当前用户打开后不知道下一步。成功标准是激活率从 35% 向 50% 靠近，
护栏是 activation_complete 后的留存不下降，观察窗口是 72 小时。
```

## 6. 实验设计

### 6.1 实验记录表

| 字段 | 内容 |
| --- | --- |
| 实验名 | 例如 `landing_hook_a_vs_b` |
| 假设 | 如果改 X，会让 Y 指标提升，因为 Z |
| 处理组 | A / B / C 各改什么 |
| 用户范围 | 新用户、某渠道、某国家、某版本 |
| 主指标 | 用来决定继续/停止 |
| 驱动指标 | 用来解释原因 |
| 护栏指标 | 不能恶化 |
| 成功线 | 继续或扩大条件 |
| 停止线 | 回滚或停止条件 |

## 7. 复盘节奏

| 节奏 | 看什么 | 目的 |
| --- | --- | --- |
| 每日 | 数据是否正常、P0 断点、异常波动 | 保证接入和执行 |
| 每 72 小时 | 素材、入口、关键路径实验 | 判断 keep / iterate / stop |

## 8. Go / Hold / Stop 闸门

### Go

满足后才加预算或扩大流量：

- 主指标达到或接近目标。
- 下游指标同向改善，例如转化和留存也变好。
- 护栏指标没有明显恶化。
- 数据口径已经稳定。
- 至少一个来源、素材或版本连续 2-3 个观察窗口表现更好。

### Hold

保持小流量学习：

- 指标方向可读，但样本不足。
- 上游改善，下游尚未确认。
- 关键数据源缺失但不阻塞小实验。
- 需要更多 cohort 才能判断。

### Stop

暂停、回滚或转向：

- 只提升点击，不提升激活、转化或留存。
- 主指标改善但护栏恶化，例如退款、投诉、崩溃、取消。
- 两轮实验后核心断点没有改善。
- 数据口径不可信，无法支撑决策。
- 获客成本明显高于可承受业务价值。

## 9. 通用看板结构

推荐每个业务做一个同样结构的看板：

| 区块 | 内容 |
| --- | --- |
| 当前状态 | Go / Hold / Stop、当前阶段、今天第一动作 |
| 北极星 | 当前值、目标、趋势 |
| 核心漏斗 | 从来源到业务价值的 4-6 步 |
| 渠道质量 | source、campaign、creative_id、country |

| 护栏 | 退款、投诉、性能、合规、成本 |

## 11. 可复制的指标字典

| 指标 | 分子 | 分母 | 时间窗口 | 数据源 | 修改目标 |
| --- | --- | --- | --- | --- | --- |
| 访问到激活率 | `activation_start` 用户数 | visit / app_open 用户数 | 日 / 72 小时 | 客户端事件 | 首屏、落地页、CTA |
| 激活完成率 | `activation_complete` 用户数 | `activation_start` 用户数 | 日 / 72 小时 | 客户端 + 服务端事件 | onboarding、默认推荐、流程长度 |
| 转化率 | `conversion` 用户数 | 激活完成用户数 | 日 / 周 | 服务端 / 支付 / CRM | offer、定价、信任证明 |
| 二次行为率 | `repeat_action` 用户数 | 激活完成用户数 | D1 / D7 | 事件 / 仓库 | 推荐、提醒、循环机制 |
| D1 留存 | 次日活跃 cohort 用户数 | 当日新用户 cohort | cohort | GA / 仓库 / 平台 | 回访理由、召回、内容更新 |
| 获客成本 | spend | conversion 或 activation_complete 用户数 | 日 / 周 | 广告平台 + 事件 | 素材、渠道、定向 |
| 质量护栏 | refund / complaint / crash 用户数 | 活跃或购买用户数 | 日 / 周 | 服务端 / 客服 / 崩溃平台 | 质量、承诺、性能 |

## 12. 产出格式

每次给业务方输出时，尽量只给这 5 件事：

1. 当前最大断点是什么。
2. 证据来自哪里，哪些结论暂时不能下。
3. 本轮应该修改什么触点。
4. 用哪个指标判断继续或停止。
5. 下一个 72 小时只做哪一个实验。

## 13. 小游戏 / 无登录产品 / GA4 补充

轻量小游戏、PWA、匿名工具和 Alpha 测试产品常见问题是：没有登录用户、没有收入事件、样本很小，但又需要每天判断增长断点。套用本模板时需要额外补这几项。

### 13.1 无登录身份口径

| 字段 | 建议 |
| --- | --- |
| `anonymous_id` | 用 GA client id、设备本地 ID 或本地存储生成的匿名 ID；不要采集姓名、邮箱、手机号 |
| `cohort_date` | 记录首次看到用户的日期，用于近似 D1 / D7 |
| `visit_count` | 本地累计访问次数，用于区分首访和回访 |
| `days_since_first_seen` | 首访至今的天数 |
| `days_since_last_seen` | 上次访问至今的天数 |

无登录场景不要把 `return_event / active_users` 当作严格 D1 留存。它只能作为回访信号；严格 cohort 需要 GA cohort、仓库或 BigQuery 事件级数据。

### 13.2 无营收阶段的 conversion 映射

前三个月增长优先时，`conversion` 可以先映射为“完成核心循环”，例如：

| 产品 | conversion 代理 |
| --- | --- |
| 轻量游戏 | 完成首局、完成第二局、完成每日挑战 |
| 工具产品 | 首次导出、首次保存、首次分享 |
| 内容产品 | 完读、完播、收藏、分享 |

文档中必须标明这是增长代理指标，不是收入或业务价值的最终口径。

### 13.3 游戏类最小事件

| 事件 | 含义 |
| --- | --- |
| `app_open` | 打开产品 |
| `mode_start` | 选择模式或入口 |
| `activation_start` | 开始第一局 / 第一关 / 首个核心任务 |
| `first_input` | 第一次有效输入 |
| `activation_complete` | 完成第一局 / 第一关 / 首个核心任务 |
| `repeat_action` | 第二局或后续核心循环 |
| `retention_return` | 跨天回访 |
| `share_click` / `share_success` | 分发动作 |
| `app_error` / `performance_summary` | 质量护栏 |

### 13.4 GA4 注意事项

- GA4 自定义维度和指标不会回填历史，必须记录创建时间。
- GA4 Data API 标准报表更适合自然日窗口；严格滚动 24 小时需要事件级导出或 BigQuery。
- 高基数字段如 `session_id`、`attempt_id`、完整 URL query 不适合注册为普通 GA4 UI 自定义维度。
- 自定义维度优先注册低基数字段：入口、模式、结果、国家、设备、难度、版本、活动、素材。
- 小样本阶段先判断事件是否打通、最大断点在哪里，再看百分比是否显著。

### 13.5 自动化复盘建议

```text
2. app_open -> activation_start -> first_input -> activation_complete -> repeat_action -> retention_return

4. share / error / performance 护栏
5. 明天只做 1-3 个动作
```
